Overview

Provide text alternatives for non-text content. Typically, this involves multimedia (images, video), and interactive components (controls, inputs). Understanding 1.1.1

Images: Know the Differnce

Images typically fall into one of two categories: Decorative and Informative. It is very important to understand the difference between these two image types in order to make a more informed decision.

  • Decorative: Images of a non-important nature. For example, icons or other visual embellishments. If a page can be understood without these images, then an alternative text value is not required.
  • Informative: Images of an important nature. These are often images that provide value to the surrounding content. If a page can't be understood or experienced in a similar way to the visual experience, then an alternative text value is required.

Images: Basic Rules

  1. All images require an alt attribute, whether they are informative or decorative. For example: <img alt="" src="...">. An image with no alt attribute value is known as a "null alt".
  2. If image is informative or contextually important to surrounding content, then it must include an alternative text value.
  3. If image is meant to invoke a feeling or emotion, then it should contain an alternative text value.
  4. If image contains text, then it must include an alternative text value. Typically verbatim, to match what is embedded in image.
  5. If image is purely decorative, then it requires no alternative text value.
  6. If image is contained within a hyperlink or button, then it must contain functional alternative text. For example: <a href="..."><img alt="My Awesome Vacation (Video)" src="..."></a>.
  7. If image within hyperlink or button contains adjacent or nearby text, then the previous rule can be ignored. For example: <a href="..."><img alt="" src="..."> <span>My Awesome Vacation (Video)</span></a>
  8. In all cases where an alternative text description is deemed necessary, avoid phrases like "Image of ..." or "Picture of ..." as these are redundant to assistive technology (AT) users.
  9. If image is deemed informative and is found to be a background image, alternaive techniques must be taken to ensure that this information is conveyed to AT users. Here it may be easier to simply use an inline image instead.

Test Procedure(s)

Review images and ensure that the aforementioned rules are met. You can use the following tools to review page images in more depth:

  • View Page Images

Accessibility Tip

Alternative text should be written in a descriptive, yet succinct manner. Think of how you might describe the image to a friend who is blind.

Note: Unless it is relevant to the image, intentionally including keywords in alternative text descriptions, for the sole benefit of SEO, is strongly discouraged. Websites are for people; not machines.

Components and Multimedia: Basic Rules

  1. All form and interactive buttons must have a descriptive value. If text is not explicit in the design of such components, then you may use various methods to provide this information. For example, if a menu or search button has no visual value, you might need to do something like this: <button aria-label="Menu"></button>
  2. Form inputs have associated text labels. Placeholder values should not be used as a stand-in for a proper label.
  3. Multimedia is identified via accessible text. For example: <video src="..." title="My Summer Vacation">...</video>
  4. Inline frames (iframe) are appropriately titled and describe contents of frame. For example: <iframe src="..." title="Job Application Form">...</iframe>

Test Procedure(s)

  • Video on a page should be manually reviewed. If the video includes dialogue or important information, then it must contain captions and/or an audio description.

Don't use presentation that relies solely on color. Understanding 1.4.1

Test Procedure(s)

  • Manually ensure that color is not used as the sole method of conveying content or distinguishing visual elements. For example, saying "Please correct all text highlighted in red", would be a failure.
  • Ensure that hyperlinks meet a 3:1 color contrast ratio between its own color and surrounding text, particularly if no underline is present. If no underline is present, an additional distinction must be provided when the link is hovered on or receives focus.
  • Test the interface under various color-blindness simulations to see if there are any potential contrast issue:

Overview

Accessible by keyboard only. Understanding 2.1.1

Test Procedure(s)

  • Tab through and engage with interactive components on the page. Anything that can be used with a pointing device, such as a mouse, must be equally usable via keyboard.
  • Ensure that interactive components on the page, such as buttons, can be used with both the Enter and Spacebar keys.
  • See where focus is brought when certain components are engaged with. For example, what happens when a button is clicked on and a modal window is shown? Will continuing to tab take you behind the modal or within the modal itself?
  • Does keyboard focus become trapped within any portion of the interface? On modals, this may be acceptable behavior, but in other scenarios it could be a keyboard trap.

Note: If focus outlines have been suppressed (which is also a failure), you can on the page to see where you are navigating.

Overview

Provide user controls for moving content. Understanding 2.2.2

Test Procedure(s)

  • Manually check to see if the page contains automatically moving content, such as a carousel or video. If such features have no mechanism to pause them, then this is a failure. Web animations, such as micro animations, are exempt from this rule, but it is strongly recommended that development leverage prefers-reduced-motion in their work.

Overview

No content flashes more than three times per second. Understanding 2.3.1

Test Procedure(s)

  • A high rate of flashing can cause potential harm, or even death, in individuals with epilepsy. If you suspect that an element on a page may be harmful, you can download and run PEAT (Photosensitive Epilepsy Analysis Tool) to see if there is any risk or reach out to the Accessibility Team to perform this test.

Note: Any concerns or failures should also be brought to the attention of Michael Spellacy.

Overview

Provide a 'Skip to Content' link. Understanding 2.4.1

Test Procedure(s)

  • Areas of a web page that may be cumbersome to navigate through via keyboard or assistive technology, must provide a mechanism to skip over that content. These "Skip" links should be available to all users when needed and not obscure other content on the page when visible. You will often find these links placed before a websites primary navigation. If it takes you a long time to tab through a certain portion of the interface, then this may be a good indication that a skip link is necessary.

Overview

Ensure keyboard focus is visible and clear. Default focus outlines are a browser feature meant to aid those with low vision, and should never be removed without an adequate replacement. If default focus outlines are undesirable, they may be redesigned or an alternative focus state can be used. However, any redesigned outline or alternative focus state must meet a 3:1 contrast ratio. Understanding 2.4.7

Test Procedure(s)

  • Tab through a page to review focus states of interactive elements. If focus state is missing or an alternative and adequate focus state is not provided, then this is a failure.
  • Be mindful of focus order. If tab stops are not in a fairly linear order, then this should be flagged as a failure.
  • An interactive element with a tabindex other than 0 will interfere with a pages natural tab order and should be considered a failure. A tabindex with -1 will not interfere with tabbing.
  • Non-interactive elements should not contain a focus state.
